# هفته سوم: مفاهیم پیشرفته شیءگرایی، مدیریت استثناها و ساختار ماژولار

این مستند خلاصه‌ای از مفاهیم کلیدی پایتون است که در طول یادگیری مفاهیم شیءگرایی پیشرفته، MRO، مدیریت خطا و ساختار پروژه جمع‌آوری شده است.

---

## ۱. اصول وراثت (Inheritance)

وراثت یکی از ارکان برنامه‌نویسی شیءگرا است که به یک کلاس (فرزند) اجازه می‌دهد ویژگی‌ها و متدهای کلاس دیگر (والد) را به ارث ببرد.

### نکات کلیدی
- **جلوگیری از تکرار کد (DRY):** ویژگی‌ها و رفتارهای مشترک در کلاس والد تعریف می‌شوند و کلاس‌های فرزند از آن استفاده می‌کنند.
- **کلاس پایه‌ی `object`:** تمام کلاس‌ها در پایتون به‌طور پیش‌فرض از کلاس `object` ارث‌بری می‌کنند.
- **توابع کاربردی برای بررسی وراثت:**
  - `isinstance(obj, ClassName)`: بررسی می‌کند آیا یک شیء نمونه‌ای از یک کلاس (یا کلاس‌های والد آن) است یا خیر.
  - `issubclass(ChildClass, ParentClass)`: بررسی می‌کند آیا یک کلاس، زیرکلاسی از کلاس دیگر است یا خیر.

---

## ۲. وراثت چندگانه و الگوریتم MRO

پایتون از **وراثت چندگانه (Multiple Inheritance)** پشتیبانی می‌کند؛ یعنی یک کلاس فرزند می‌تواند به‌طور هم‌زمان از چند کلاس والد ارث‌بری کند.

### ترتیب پیمایش متدها (Method Resolution Order - MRO)
وقتی یک متد یا ویژگی روی یک شیء فراخوانی می‌شود، پایتون بر اساس یک ترتیب خطی و مشخص به نام **MRO** جست‌وجو را انجام می‌دهد:
1. ابتدا خود کلاس فرزند را بررسی می‌کند.
2. سپس طبق ترتیب ذکرشده در پرانتز تعریف کلاس، کلاس‌های والد را از چپ به راست پیمایش می‌کند.
3. در نهایت به کلاس ریشه (`object`) می‌رسد.

- **مشاهده MRO:** با استفاده از `ClassName.__mro__` یا `ClassName.mro()` می‌توان ترتیب دقیق فراخوانی را مشاهده کرد.

## ۳. تابع `super()` و حل «مشکل الماس»

در وراثت چندگانه، فراخوانی مستقیم متدهای والد (مانند `ParentClass.__init__(self)`) ممکن است باعث اجرای چندباره‌ی سازنده‌ی یک کلاس پایه شود (این مشکل به **مشکل الماس** یا Diamond Problem معروف است).

### نقش `super()`
- **ارث‌بری تعاونی (Cooperative Inheritance):** تابع `super()` به جای فراخوانی مستقیم والد، کلاس **بعدی** را در زنجیره MRO پیدا کرده و متد آن را اجرا می‌کند.
- **تضمین اجرای تک‌باره:** این مکانیزم تضمین می‌کند که در یک سلسله‌مراتب پیچیده، متد هر کلاس والد دقیقاً یک بار و به ترتیب صحیح اجرا شود.
- **انتقال آرگومان‌ها با `**kwargs`:** هنگام استفاده از `super()` در یک ساختار پیچیده، بهتر است از آرگومان‌های کلیدواژه‌ای (`**kwargs`) استفاده شود تا هر کلاس پارامترهای مربوط به خود را بردارد و باقی‌مانده را به والد بعدی در زنجیره MRO ارسال کند.

## ۴. مدیریت استثناها (Exception Handling)

استثناها اشیائی در پایتون هستند که در صورت بروز خطا پرتاب (raise) می‌شوند. برای جلوگیری از توقف ناگهانی برنامه، از ساختار مدیریت استثنا استفاده می‌کنیم.

### ساختار کامل `try...except...else...finally`

```python
try:
    # کدی که احتمال بروز خطا در آن وجود دارد
    result = 10 / 2
except ZeroDivisionError as e:
    # این بلوک فقط در صورت بروز خطای تقسیم بر صفر اجرا می‌شود
    print(f"خطا: {e}")
except (TypeError, ValueError) as e:
    # مدیریت چند نوع خطا به‌صورت یکجا
    print(f"خطای نوع یا مقدار: {e}")
else:
    # این بلوک فقط در صورت «عدم بروز هیچ‌گونه خطا» در بلوک try اجرا می‌شود
    print(f"عملیات موفقیت‌آمیز بود. نتیجه: {result}")
finally:
    # این بلوک «همیشه» اجرا می‌شود (چه خطا رخ دهد چه ندهد)
    # مناسب برای آزادسازی منابع مانند بستن فایل یا اتصال شبکه
    print("پایان عملیات.")
```

**چرا از `else` استفاده کنیم؟**
جداسازی کدی که پس از موفقیت `try` اجرا می‌شود، خوانایی را بالا می‌برد و از مدیریت ناخواسته خطاهای منطقی توسط بلوک `except` جلوگیری می‌کند.

## ۵. پرتاب و تعریف استثناهای سفارشی

### کلیدواژه‌ی `raise`
برای پرتاب دستی یک استثنا استفاده می‌شود.
```python
raise ValueError("یک پیام خطای مشخص")
```
- استفاده از `raise` به‌تنهایی در بلوک `except`، همان استثنای دریافت‌شده را مجدداً پرتاب (Re-raise) می‌کند تا در سطوح بالاتر برنامه مدیریت شود.

### تعریف استثنای اختصاصی (Custom Exception)
برای ایجاد خطاهای معنادار و مشخص برای برنامه، می‌توان کلاس‌های خطای سفارشی تعریف کرد که از `Exception` (یا زیرکلاس‌های آن) ارث‌بری می‌کنند.
```python
class CustomApplicationError(Exception):
    def __init__(self, message, error_code):
        super().__init__(message)
        self.error_code = error_code

# نحوه استفاده
try:
    raise CustomApplicationError("خطایی در منطق برنامه رخ داد.", 500)
except CustomApplicationError as e:
    print(f"کد خطا: {e.error_code}, پیام: {e}")
```

## ۶. ماژولارنویسی و ساختار فایل‌ها

برای نگهداری و مقیاس‌پذیری کد، برنامه‌ها به فایل‌ها (ماژول‌ها) و پوشه‌ها (بسته‌ها) تقسیم می‌شوند.

### مفاهیم پایه
- **ماژول (Module):** هر فایل پایتون با پسوند `.py` یک ماژول است.
- **بسته (Package):** هر دایرکتوری که حاوی یک فایل `__init__.py` (حتی خالی) باشد، توسط پایتون به عنوان یک بسته شناخته می‌شود.

### نحوه Import کردن
- `import module_name`: دسترسی از طریق `module_name.function`.
- `from module_name import function_name`: دسترسی مستقیم به `function_name`.
- `import module_name as alias`: تغییر نام ماژول برای استفاده آسان‌تر.
- `from module_name import *`: وارد کردن تمام تعاریف عمومی (**توصیه نمی‌شود**).

### کنترل اجرای مستقیم با `__name__`
متغیر سراسری `__name__` مشخص می‌کند که یک فایل چگونه اجرا شده است:
- اگر فایل **مستقیماً** اجرا شود: `__name__` برابر با `"__main__"` خواهد بود.
- اگر فایل به عنوان ماژول **وارد (import)** شود: `__name__` برابر با نام آن ماژول (نام فایل) خواهد بود.

این الگو برای قرار دادن کدهای تست یا نمونه در خود ماژول بسیار کاربردی است.
```python
def my_function():
    print("Hello from my_function")

if __name__ == "__main__":
    # این کد فقط زمانی اجرا می‌شود که این فایل را مستقیم اجرا کنیم
    print("فایل به صورت مستقیم اجرا شد.")
    my_function()
```

### کشینگ ماژول (Module Caching)
- پایتون ماژول‌های وارد شده را **کش (cache)** می‌کند.
- اگر یک ماژول برای بار دوم `import` شود، پایتون از نسخه‌ی کش‌شده استفاده می‌کند و کد ماژول دوباره اجرا نمی‌شود.
- این یعنی تغییرات در متغیرهای یک ماژول در طول یک جلسه (session) برنامه باقی می‌ماند.

## ۷. روش‌های دیباگ کردن (Debugging)

دیباگ کردن بخش جدایی‌ناپذیر از فرایند توسعه است. در ادامه چند روش کاربردی معرفی می‌شود.

### ۱. استفاده از تابع `print()`
- **ساده‌ترین روش:** چاپ مقادیر متغیرها در نقاط مختلف کد برای ردیابی جریان اجرای برنامه.
- **نکته:** پس از رفع مشکل، `print`های اضافی را حتماً حذف کنید تا کد شلوغ نشود.

### ۲. استفاده از `assert`
- **بررسی پیش‌شرط‌ها:** `assert` یک شرط را بررسی می‌کند و اگر `False` باشد، یک خطای `AssertionError` ایجاد کرده و برنامه را متوقف می‌کند.
- این دستور برای اطمینان از صحت شرایط و فرضیات در نقاط حساس کد بسیار مفید است.
```python
def divide(a, b):
    assert b != 0, "Cannot divide by zero"
    return a / b
```

### ۳. تست پس از هر تغییر کوچک
- **توسعه تدریجی:** به جای نوشتن حجم زیادی از کد و سپس تست آن، هر بخش کوچک را پس از پیاده‌سازی، فوراً تست کنید.
- این کار پیدا کردن منبع خطا را بسیار آسان‌تر می‌کند.
- ورودی‌های مختلف را تست کنید: مقادیر معمولی، لبه‌ای (صفر، منفی، خالی) و نامعتبر.

### ۴. خواندن و جست‌وجوی پیام خطا
- **Traceback مهم است:** پیام خطای پایتون (Traceback) اطلاعات دقیقی در مورد نوع خطا و محل وقوع آن می‌دهد.
- **جست‌وجو کنید:** اگر پیام خطا واضح نیست، متن دقیق آن را در موتورهای جست‌وجو یا ابزارهای هوش مصنوعی جست‌وجو کنید. به احتمال زیاد شخص دیگری قبلاً با این مشکل مواجه شده است.

### ۵. استفاده از کاغذ و قلم (و استراحت!)
- **سازماندهی افکار:** رسم فلوچارت یا نوشتن مراحل اجرای برنامه روی کاغذ می‌تواند به درک منطق پیچیده و پیدا کردن خطا کمک کند.
- **فاصله بگیرید:** گاهی اوقات کمی استراحت و فاصله گرفتن از مشکل، بهترین راه برای پیدا کردن راه‌حل با ذهنی بازتر است.

### ۶. استفاده از دیباگرها (Debugger)
- **ابزار قدرتمند:** دیباگرها به شما اجازه می‌دهند اجرای برنامه را در نقاط دلخواه (Breakpoint) متوقف کرده، کد را خط‌به‌خط اجرا کنید و مقادیر متغیرها را در هر لحظه مشاهده نمایید.
- **انواع دیباگر:**
  - **`pdb` (Python Debugger):** دیباگر داخلی و خط فرمانی پایتون.
  - **دیباگرهای گرافیکی:** ابزارهای تعبیه‌شده در IDEها مانند PyCharm و VS Code که رابط کاربری راحت‌تری دارند.
```shell
# اجرای یک اسکریپت با دیباگر pdb
python -m pdb your_script.py
```
آشنایی با `pdb` برای دیباگ سریع در محیط‌های مختلف بسیار مفید است.
