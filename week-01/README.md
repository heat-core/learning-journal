یادداشت‌های گیت - هفته اول

توی این فایل، یادداشت‌ها و چیزهایی که درباره گیت (Git) یاد گرفتم رو مرتب و دسته‌بندی کردم تا همیشه بتونم بهشون سر بزنم.

۱. تنظیمات اولیه و شروع کار (Setup & Init)

تنظیم مشخصات هویتی: اولین کاری که بعد نصب گیت کردم، مشخص کردن ایمیل و نام کاربریم بود:

git config --global user.email "me@example.com"
git config --global user.name "My name"


ساخت مخزن جدید:

git init: یه مخزن (repository) خالی جدید توی پوشه فعلی میسازه.

مشاهده فایل‌ها:

ls -la: لیست همه فایل‌ها و پوشه‌ها (حتی مخفی‌ها) رو نشون میده تا ببینیم پوشه موردنظر وجود داره یا نه. (یادآوری: این دستور صرفاً همه فایل‌ها رو لیست میکنه و چک کردن هوشمند پوشه نیست).

ls -l .git/: محتویات داخل پوشه مخفی .git رو نشون میده.

۲. مدیریت وضعیت فایل‌ها و ثبت تغییرات (Staging & Committing)

اضافه کردن به Staging Area (آماده‌سازی برای ثبت):

git add XXX: فایل مشخص‌شده رو ردیابی میکنه و میبره به staging area.

git add README: فایل README رو به staging area اضافه میکنه.

git add .: همه فایل‌های تغییرکرده رو یکجا stage میکنه.

بررسی وضعیت:

git status: وضعیت فعلی working tree (فولدر پروژه) و تغییراتی که منتظر کامیت هستن رو نشون میده.

کامیت کردن (ثبت نهایی تغییرات):

git commit: تغییرات موجود توی staging area رو ثبت میکنه و میفرسته داخل پوشه .git.

git commit -m "This is my second commit.": مستقیم با نوشتن پیام توی خط فرمان کامیت میکنه.

git commit -a: فایل‌هایی که قبلاً ردیابی شدن و الان تغییر کردن رو خودکار stage و کامیت میکنه (روی فایل‌های جدید یا untracked کار نمیکنه).

راهنمای نوشتن پیام Commit مناسب

یه پیام کامیت استاندارد معمولاً دو بخش داره:

خلاصه (Summary): خط اول باید یه خلاصه کوتاه (حداکثر ۵۰ کاراکتر) باشه که مثل عنوان عمل میکنه.

توضیحات کامل (Description): خطوط بعدی (هر خط زیر ۷۲ کاراکتر) که جزییات تغییرات، ارجاع به باگ‌ها یا لینک‌های کمکی رو شامل میشه.

۳. مشاهده تاریخچه و تفاوت‌ها (History & Diff)

ویرایش سریع فایل:

nano README: فایل README رو توی ادیتور نانو باز میکنه.

بررسی تفاوت‌ها (Diff):

git diff: تفاوت فایل‌های تغییر کرده ولی stage نشده رو نشون میده.

git diff README: تفاوت‌های فایل README رو نشون میده.

git diff README README1: تفاوت‌های بین دو فایل README و README1 رو مقایسه میکنه.

git diff --staged (یا همان git diff --cached): تفاوت فایل‌های stage شده رو نسبت به آخرین کامیت نشون میده.

بررسی تاریخچه کامیت‌ها:

git log: تاریخچه کامیت‌ها رو نشون میده.

git log -p: علاوه بر تاریخچه، کدهای تغییرکرده (Patch) هر کامیت رو هم نشون میده.

git log --graph: نمودار درختی کامیت‌ها و مرج‌ها رو به صورت متنی (ASCII) نشون میده.

git log --oneline: هر کامیت رو خلاصه و توی یک خط نشون میده.

نمایش جزییات یک آبجکت:

git show: جزییات کامیت‌ها، تگ‌ها یا آبجکت‌های گیت رو نشون میده.

۴. دستورات پیشرفته‌تر و مدیریت فایل‌ها

git add -p: اجازه میده تغییرات فایل‌ها رو به صورت بخش‌بخش (تعاملی) بررسی کنی و برای کامیت بفرستی.

git mv: برای جابه‌جا کردن یا تغییر نام فایل‌ها و پوشه‌ها استفاده میشه (مثل دستور mv در لینوکس).

git rm: فایل رو از working tree و سیستم ردیابی گیت حذف میکنه.

کار با فایل .gitignore

برای اینکه گیت بعضی فایل‌ها (مثل فایل‌های تنظیمات شخصی، متادیتا یا فایل‌های لوکال) رو نادیده بگیره و ردیابی نکنه، از این فایل استفاده میکنیم.

نکات نوشتن:

خطی که با # شروع بشه کامنته.

کاراکتر / جداکننده پوشه‌هاست.

نمونه الگوهای آماده گیت‌اینور

۵. لغو تغییرات و بازگشت به عقب (Undoing & Reset)

تعویض برنچ یا بازگردانی:

git checkout <branch>: شاخه فعال رو تغییر میده. مثلاً اگه بخوای کدهای جدید رو از برنچ main بگیری (pull کنی)، اول با این دستور میری روی اون:

git checkout main


و بعد دوباره میتونی برگردی روی برنچ خودت:

git checkout <branch>


خارج کردن از حالت Stage:

اگه کل فایل‌ها رو با دستور git add . اشتباهی stage کردی ولی هنوز آماده نبودی، با دستور زیر تغییرات رو از استیج خارج کن بدون اینکه کدهات خراب بشن:

git reset


(نکته خیلی مهم: هارد ریست با دستور git reset --hard خطرناکه چون تغییرات محلیِ ذخیره نشده رو کاملاً پاک میکنه).

ذخیره موقت تغییرات (راه امن‌تر):

git stash: تغییرات فعلی رو موقتاً یه گوشه ذخیره میکنه تا بعداً بتونی بهشون برگردی (روشی خیلی امن‌تر از ریست سخت).

اصلاح آخرین کامیت:

git commit --amend: برای اصلاح پیام آخرین کامیت یا اضافه کردن فایل‌های جا افتاده استفاده میشه. (تذکر مهم: این دستور کامیت قبلی رو بازنویسی میکنه؛ پس روی کامیت‌هایی که قبلاً push شدند نباید استفاده بشه).

برگشت امن با کامیت معکوس:

git revert: یک کامیت جدید میسازه که کارش دقیقاً برعکس کردن تغییرات کامیت قبلیه. این دستور تاریخچه رو خراب نمیکنه و امن‌تر از ریست هست.

برای جزییات بیشتر مراجعه کن به: README.md یا لینک Git Basics – Undoing Things

۶. مکانیزم ذخیره‌سازی گیت (SHA-1)

گیت برای شناسایی و ذخیره داده‌ها از الگوریتم SHA-1 استفاده میکنه. این تابع هش برای هر کامیت و فایل یک شناسه منحصربه‌فرد (اثر انگشت دیجیتال) میسازه که دو کاربرد اصلی داره:

تایید دست‌نخورده بودن محتوای فایل (امضای دیجیتال)

شناسایی و آدرس‌دهی فایل‌ها و کامیت‌ها (مثلاً استفاده از هش در دستور git revert)

این هش‌ها رو میتونی توی خروجی دستور git log ببینی و ازشون برای اشاره به کامیت‌های مختلف استفاده کنی.

۷. مدیریت شاخه‌ها (Branching & Merging)

git branch: لیست برنچ‌های موجود رو نشون میده.

git branch <name>: یه برنچ جدید میسازه.

git branch -d <name>: برنچ رو حذف میکنه.

git branch -D <name>: برنچ رو به زور (اجباری) حذف میکنه.

git checkout -b <new-branch>: یه برنچ جدید میسازه و مستقیم میره روی اون.

git merge <branch>: تغییرات برنچ مشخص‌شده رو با برنچ فعلی ادغام میکنه.

git merge --abort: اگه موقع مرج تعارضی (Conflict) پیش اومد، پروسه رو لغو میکنه و پروژه رو به حالت قبل مرج برمیگردونه.

اصطلاحات کلیدی ادغام (Merge)

Fast-forward merge: زمانی اتفاق میفته که برنچ اصلی تغییر جدیدی نداشته و گیت فقط اشاره‌گر (Pointer) رو جلو میبره.

HEAD: اشاره‌گری که به آخرین کامیتِ برنچ فعال (شاخه‌ای که الان توش هستیم) اشاره میکنه.

Master / Main: شاخه اصلی پروژه که معمولاً کدهای نهایی و بدون باگ اونجا قرار میگیرن. (توضیح: توی مخازن جدید معمولاً main جایگزین master شده).

Merge conflict (تعارض): وقتی پیش میاد که روی یک خط مشخص از یک فایل، در دو برنچ مختلف تغییرات متفاوتی اعمال شده باشه و گیت نتونه خودکار تشخیص بده کدوم درسته.

Rollback: برگردوندن کدهای نرم‌افزار به نسخه پایدار قبلی.

Three-way merge: زمانی که شاخه‌ها از هم فاصله گرفتن و امکان Fast-forward نیست، گیت سه نقطه رو مقایسه میکنه (کد شاخه اول، کد شاخه دوم، و آخرین کامیت مشترک آن‌ها) تا یک کامیت ادغام جدید بسازه.

۸. کار با مخازن ریموت (Remote Repositories)

سایت‌های معروف برای میزبانی ریموت: GitHub, GitLab, BitBucket.

git clone URL: کپی کردن کامل یک مخزن ریموت روی سیستم لوکال.

git push: فرستادن کامیت‌های لوکال به مخزن ریموت.

git pull: دریافت و ادغام آخرین تغییرات از ریموت روی سیستم لوکال.

git remote: مدیریت مخازن ریموت متصل به پروژه.

git remote -v: نمایش لیست ریموت‌ها همراه با آدرس URL آن‌ها.

git remote show <name>: نمایش جزییات یک ریموت خاص.

git remote update: دریافت آخرین وضعیت تمام ریموت‌ها بدون ادغام کدهای لوکال.

git fetch: دانلود کردن فایل‌ها و کامیت‌های جدید از ریموت بدون اینکه روی کدهای فعلی شما ادغامشون کنه.

git branch -r: نمایش لیست برنچ‌های موجود در ریموت.

نکات حل تعارض‌ها (Conflicts)

شرط اول ثبت کد: تا وقتی تداخل‌ها (Conflicts) توی سیستم لوکال حل نشن، گیت اجازه ثبت کامیت جدید یا push کردن رو نمیده. وقتی تعارض‌ها رو کامل حل کردی، میتونی تغییرات رو push کنی و اون‌ها رو توی یک Pull Request ادغام کنی.

پیام‌های خطای گیت رو موقع pull یا push با دقت بخونید؛ مخصوصاً خطاهایی که میگن نمیتونید تغییرات محلی رو push کنید چون نسخه ریموت جلوتره.

برای حل تعارض‌ها بعد از کلون کردن مخزن روی سیستم خودت، میتونی از خط فرمان یا GitHub Desktop برای push کردن تغییرات به برنچ خودت در گیت‌هاب استفاده کنی.

یک کار تیمی قوی: قبل از ادغام نهایی روی شاخه اصلی (main/master)، کدهاتون رو روی برنچ خودتون push کنید تا بقیه همکارها بتونن کد رو ببینن، بررسی و تست کنن و بعد از تایید نهایی مرج بشه.

میشه از git rebase استفاده کرد تا تغییرات برنچ شما روی آخرین نسخه شاخه اصلی بازنویسی بشه و بعد برنچ فیچر رو دوباره به شاخه اصلی مرج کرد تا تاریخچه کاملاً خطی و تمیز بمونه.
